## 一、概述

### 1.1 背景

技术选型已全部确定（LLM、RAG框架、向量数据库、OCR等），在正式启动MVP开发前，需要完成技术验证、架构设计和环境搭建等准备工作，确保开发过程顺畅。

### 1.2 目标

1. **验证技术可行性**：通过POC验证核心技术在实际场景中的表现
2. **设计系统架构**：清晰、可扩展的架构设计
3. **搭建开发环境**：统一、规范的开发环境
4. **制定开发规范**：团队协作和质量保证标准
5. **规避技术风险**：提前发现和解决潜在问题

### 1.3 预期成果

- ✅ 技术可行性验证报告
- ✅ 完整的系统架构设计文档
- ✅ 可运行的开发环境
- ✅ 规范的项目代码结构
- ✅ 详细的开发规范文档

```yaml

验收标准:
  - 查询延迟: <100ms
  - Top-5准确率: >85%
  - 并发能力: >10 QPS

验证步骤:
  1. 安装PostgreSQL 15 + pgvector扩展
  2. 准备测试数据集（100条法律条文）
  3. 使用LlamaIndex建立索引
  4. 实现基础查询功能
  5. 测试检索性能和准确率
  6. 记录测试数据

预期产出:
  - POC代码示例

  - 准确率评估报告
```

#### 2.2.2 数据库设计

**核心表结构**：

```sql
-- 用户表
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    username VARCHAR(50) UNIQUE NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL,
    hashed_password VARCHAR(255) NOT NULL,
    full_name VARCHAR(100),
    role VARCHAR(20) NOT NULL, -- intern, associate, partner
    is_active BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

-- 案件表
CREATE TABLE cases (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID REFERENCES users(id),
    case_number VARCHAR(100),
    case_name VARCHAR(200) NOT NULL,
    crime_type VARCHAR(100),
    case_type VARCHAR(50), -- 侦查阶段、审查起诉、一审
    status VARCHAR(20) DEFAULT 'active', -- active, closed, archived
    summary TEXT,
    metadata JSONB, -- 灵活存储其他信息
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

-- 文档表
CREATE TABLE documents (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    case_id UUID REFERENCES cases(id),
    file_name VARCHAR(255) NOT NULL,
    file_type VARCHAR(20), -- pdf, image, docx
    file_path TEXT, -- 存储路径
    file_size BIGINT,
    doc_type VARCHAR(50), -- 讯问笔录、询问笔录、鉴定意见等
    page_count INT,
    ocr_status VARCHAR(20), -- pending, processing, completed, failed
    uploaded_at TIMESTAMP DEFAULT NOW()
);

-- 证据表
CREATE TABLE evidences (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    case_id UUID REFERENCES cases(id),
    document_id UUID REFERENCES documents(id),
    evidence_type VARCHAR(50), -- 7大证据类型
    content TEXT,
    summary TEXT,
    page_number INT,
    metadata JSONB,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 向量表（用于RAG检索）
CREATE TABLE legal_documents (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    doc_type VARCHAR(50), -- law, case, regulation
    title VARCHAR(500),
    content TEXT,
    metadata JSONB,
    embedding vector(1536), -- OpenAI embedding维度
    created_at TIMESTAMP DEFAULT NOW()
);

-- 矛盾点表
CREATE TABLE contradictions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    case_id UUID REFERENCES cases(id),
    evidence1_id UUID REFERENCES evidences(id),
    evidence2_id UUID REFERENCES evidences(id),
    contradiction_type VARCHAR(50),
    description TEXT,
    severity VARCHAR(20), -- high, medium, low
    created_at TIMESTAMP DEFAULT NOW()
);

-- 索引
CREATE INDEX idx_cases_user_id ON cases(user_id);
CREATE INDEX idx_evidences_case_id ON evidences(case_id);
CREATE INDEX idx_evidences_type ON evidences(evidence_type);
CREATE INDEX idx_documents_case_id ON documents(case_id);

-- 向量索引（pgvector）
CREATE INDEX idx_legal_docs_embedding ON legal_documents
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 100);
```

**ER图**（实体关系）：
```
users (1) ----< (N) cases (1) ----< (N) documents
    |                               |
    |                               |
    +------------------ (N) evidences
                            |
                            +----< contradictions
```

#### 2.2.3 API设计

**RESTful API规范**：

```yaml
认证相关:
  POST   /api/v1/auth/register      # 用户注册
  POST   /api/v1/auth/login         # 用户登录
  POST   /api/v1/auth/logout        # 用户登出
  GET    /api/v1/auth/me            # 获取当前用户信息

案卷管理:
  GET    /api/v1/cases              # 案件列表
  POST   /api/v1/cases              # 创建案件
  GET    /api/v1/cases/{id}         # 案件详情
  PUT    /api/v1/cases/{id}         # 更新案件
  DELETE /api/v1/cases/{id}         # 删除案件
  POST   /api/v1/cases/{id}/upload  # 上传案卷
  GET    /api/v1/cases/{id}/documents # 文档列表

文档处理:
  POST   /api/v1/documents/parse    # 解析文档
  GET    /api/v1/documents/{id}/status # 解析状态
  GET    /api/v1/documents/{id}/content # 文档内容

证据分析:
  GET    /api/v1/cases/{id}/evidences       # 证据列表
  POST   /api/v1/cases/{id}/evidences/classify # 证据分类
  GET    /api/v1/cases/{id}/contradictions  # 矛盾点列表
  POST   /api/v1/cases/{id}/contradictions/detect # 识别矛盾点

RAG查询:
  POST   /api/v1/rag/query            # RAG查询
  POST   /api/v1/rag/legal/search      # 法律条文检索
  POST   /api/v1/rag/case/search       # 判例检索
  POST   /api/v1/rag/evidence/search   # 证据语义搜索

质证辅助:
  GET    /api/v1/cases/{id}/impeachment-points # 质证要点
  POST   /api/v1/cases/{id}/impeachment/generate # 生成质证建议
```

### 3.2 开发规范制定

#### 3.2.1 代码规范

**Python代码规范**：
```yaml
代码风格:
  - 遵循PEP 8规范
  - 使用black格式化（line-length=88）
  - 使用ruff进行linting（替代flake8+pylint）
  - 使用mypy进行类型检查

命名规范:
  - 类名: PascalCase (UserService)
  - 函数名: snake_case (get_user)
  - 常量: UPPER_CASE (MAX_RETRY)
  - 私有函数: _leading_underscore (_internal_func)

类型注解:
  - 所有函数必须添加类型注解
  - 使用Pydantic进行数据验证
  - 复杂类型使用TypeVar

示例:
from typing import List, Optional
from pydantic import BaseModel

class UserCreate(BaseModel):
    username: str
    email: str
    password: str

async def create_user(user: UserCreate) -> User:
    """创建用户"""
    pass
```

**TypeScript代码规范**：
```yaml
代码风格:
  - 使用ESLint + Prettier
  - 启用strict模式
  - 使用单引号
  - 分号必须

命名规范:
  - 组件: PascalCase (UserList)
  - 函数: camelCase (getUserList)
  - 常量: UPPER_CASE (API_URL)
  - 接口: IPrefix (IUserService)

类型定义:
  - 避免使用any
  - 使用interface定义对象类型
  - 使用type定义联合类型

示例:
interface User {
  id: string;
  name: string;
  email: string;
}

const getUserList = async (): Promise<User[]> => {
  const response = await fetch('/api/users');
  return response.json();
};
```

## 六、关键决策点

### 6.1 团队组建决策

**问题**：是否需要招聘/组建团队？

**选项A：个人开发**
- 适用场景：资金有限、时间充裕
- 优势：成本低、决策快
- 劣势：开发慢、技能有限
- 建议：MVP阶段可行，成长期需要团队

**选项B：小团队开发（推荐）**
- 适用场景：有资金、追求速度
- 团队配置：
  - 1个AI工程师（全职）
  - 1个后端工程师（全职）
  - 1个前端工程师（全职或兼职）
- 成本：约2-3万/月
- 优势：开发快、专业分工
- 建议：MVP阶段推荐

### 6.2 开发模式决策

**选项A：远程协作**
- 工具：GitHub + Discord/飞书
- 优势：成本低、人才广
- 劣势：沟通成本高
- 建议：适合有远程经验的团队

**选项B：现场开发**
- 要求：办公空间
- 优势：沟通顺畅
- 劣势：成本高
- 建议：适合本地团队

**选项C：混合模式（推荐）**
- 核心：每周2-3天现场
- 其他：远程办公
- 优势：平衡成本和沟通
- 建议：适合大多数情况

### 6.3 MVP范围调整

基于POC结果，可能需要调整：

**情况1：某个技术验证失败**
→ 启用备选方案
→ 重新评估技术选型
→ 调整开发计划

**情况2：开发资源不足**
→ 削减非核心功能
→ 优先P0功能
→ 延后P1/P2功能

**情况3：时间紧张**
→ 聚焦核心价值
→ 简化UI/UX
→ MVP最小化
