## 一、调研背景

### 1.1 为什么需要RAG

SuperYou作为刑事辩护AI助手，核心价值在于对法律知识库的精准检索和证据的智能分析。RAG（Retrieval-Augmented Generation，检索增强生成）技术是实现这一目标的关键：

**核心需求**:
1. **法律条文检索**: 从海量法律条文中精准检索相关条款
2. **判例检索**: 从历史判例中找到相似案例作为参考
3. **证据语义搜索**: 基于语义理解检索相关证据材料
4. **质证知识库**: 检索质证要点和法律依据
5. **程序知识查询**: 快速查询刑事诉讼程序相关问题

**RAG的价值**:
- ✅ 提供准确的上下文信息，减少LLM幻觉
- ✅ 利用专业知识库，提高回答可信度
- ✅ 支持溯源，用户可以验证信息来源
- ✅ 降低成本，减少对昂贵LLM的依赖
- ✅ 符合法律合规要求，引用明确出处

### 1.2 调研目标

**主要目标**:
1. 明确项目对RAG的具体需求（功能、性能、成本）
2. 评估主流RAG框架的适用性（LlamaIndex、LangChain、自研）
3. 选择最适合MVP阶段的技术方案
4. 制定技术演进路线（MVP → 成长期 → 规模化）

**决策依据**:
- 开发效率（时间成本）
- 技术成熟度（稳定性）
- 功能完整性（是否满足需求）
- 学习曲线（团队能力）
- 扩展性（长期考虑）
- 社区支持（问题解决）

## 二、项目RAG需求分析

### 2.1 功能需求

#### 需求1: 法律条文检索 (P0)

**场景描述**:
用户输入："非法证据排除的法律依据是什么？"
系统需要：检索《刑事诉讼法》及相关司法解释中关于非法证据排除的条款，并准确引用。

**技术要求**:
- 检索类型: 精确匹配 + 语义搜索
- 数据量: ~5000条法律条文（刑法、刑诉法、司法解释等）
- 响应时间: <2秒
- 准确率: Top-5准确率 >90%
- 引用完整性: 必须包含法律名称、条款号、发布机关、生效日期

**输出示例**:
```
根据《中华人民共和国刑事诉讼法》第五十六条规定：
"采用刑讯逼供等非法方法收集的犯罪嫌疑人、被告人供述
和采用暴力、威胁等非法方法收集的证人证言、被害人陈述，
应当予以排除。"
```

#### 需求2: 判例检索 (P0)

**场景描述**:
用户输入："类似的危险驾驶案例怎么判决的？"
系统需要：从裁判文书中检索相似案例，提供判决结果和裁判要点。

**技术要求**:
- 检索类型: 语义搜索 + 关键词匹配
- 数据量: 10万+份裁判文书（初始1万份，逐步扩展）
- 响应时间: <3秒
- 相似度准确率: >80%
- 支持多维度检索（罪名、量刑情节、判决结果）

#### 需求3: 证据语义搜索 (P0)

**场景描述**:
用户输入："所有关于案发时间的证人证言"
系统需要：从案卷材料中检索所有提到案发时间的证人证言片段。

**技术要求**:
- 检索类型: 语义搜索 + 元数据过滤
- 数据量: 单个案卷100-500份证据
- 响应时间: <1秒
- 粒度: 段落级别检索
- 上下文窗口: 返回相关段落的前后文

#### 需求4: 质证知识库 (P1)

**场景描述**:
系统分析证据后，需要从质证知识库中检索相关的质证要点和质证技巧。

**技术要求**:
- 检索类型: 分类检索 + 关键词匹配
- 数据量: ~1000条质证要点
- 响应时间: <1秒
- 准确率: >85%

#### 需求5: 程序知识查询 (P2)

**场景描述**:
用户提问："侦查阶段的律师权利有哪些？"
系统需要：检索刑事诉讼程序相关知识并回答。

**技术要求**:
- 检索类型: FAQ匹配 + 语义搜索
- 数据量: ~500个常见问题
- 响应时间: <1秒
- 准确率: >90%

### 2.2 技术需求

#### 2.2.1 向量检索需求

| 指标 | MVP阶段 | 成长期 | 规模化 |
|------|---------|--------|--------|
| 数据量 | 10万文档 | 100万文档 | 1000万+文档 |
| 向量维度 | 768-1536 | 768-1536 | 768-1536 |
| 查询延迟 | <100ms | <200ms | <500ms |
| 并发查询 | 10 QPS | 50 QPS | 200 QPS |
| 召回率 | >90% | >90% | >90% |

#### 2.2.2 混合检索需求

**检索策略**:
1. **稠密检索**（Dense Retrieval）: 向量语义搜索
   - Embedding模型: text-embedding-3-small / bge-m3
   - 相似度计算: 余弦相似度

2. **稀疏检索**（Sparse Retrieval）: 关键词匹配
   - BM25算法
   - 全文检索（PostgreSQL tsvector）

3. **重排序**（Reranking）: 优化检索结果
   - Cross-Encoder模型
   - LLM辅助重排序

#### 2.2.3 RAG模式需求

**需要的RAG模式**:

1. **基础RAG** (Basic RAG)
   - 场景: 简单问答、知识库查询
   - 流程: 查询 → 检索 → 增强 → 生成

2. **上下文RAG** (Contextual RAG)
   - 场景: 证据分析、矛盾识别
   - 流程: 查询 → 检索多个文档 → 合并上下文 → 生成

3. **多跳RAG** (Multi-Hop RAG)
   - 场景: 复杂法律推理
   - 流程: 多轮检索 → 逐步推理 → 最终生成
   - **优先级**: P2（成长期考虑）

4. **聚合RAG** (Aggregated RAG)
   - 场景: 综合多个证据片段
   - 流程: 并行检索 → 结果聚合 → 生成
   - **优先级**: P0

#### 2.2.4 数据更新需求

| 数据类型 | 更新频率 | 更新方式 |
|---------|---------|---------|
| 法律条文 | 季度 | 全量更新 |
| 判例文书 | 每月 | 增量更新 |
| 证据材料 | 实时 | 用户上传 |
| 质证知识库 | 月度 | 人工更新 |
| 程序知识 | 季度 | 人工更新 |

### 2.3 性能需求

| 指标 | 要求 | 说明 |
|------|------|------|
| 端到端响应时间 | <3秒 | 从用户提问到显示答案 |
| 检索延迟 | <500ms | 向量检索时间 |
| LLM生成时间 | <2秒 | 生成答案时间 |
| 并发支持 | 50 QPS | 同时支持50个查询 |
| 可用性 | >99.5% | 系统可用性 |

### 2.4 成本需求

**MVP阶段成本控制**:
- 向量数据库: 0元/月（使用pgvector）
- Embedding API: <100元/月（按量计费）
- 检索开销: 包含在数据库成本中

**预期效果**:
- 通过RAG减少60%的LLM调用成本
- 提高答案准确性20%+

## 三、候选技术方案

### 3.1 方案A: LlamaIndex

**简介**: 专注于RAG的数据框架，提供高级抽象。

**优势**:
- ✅ RAG功能完整，文档质量高
- ✅ API简洁，学习曲线较平缓
- ✅ 支持多种数据源（PDF、数据库、API）
- ✅ 内置多种检索策略
- ✅ 支持流式输出和结构化输出
- ✅ 活跃的社区和丰富的示例

**劣势**:
- ⚠️ 框架依赖，增加一定复杂度
- ⚠️ 高级功能需要学习成本
- ⚠️ 定制化可能受限

**适用场景**: 中等复杂度RAG应用，需要快速开发

**学习成本**: 3-5天

### 3.2 方案B: LangChain

**简介**: 通用LLM应用框架，包含RAG功能。

**优势**:
- ✅ 生态最完善，社区最大
- ✅ 功能丰富，支持复杂的Agent应用
- ✅ 大量的集成和工具
- ✅ 文档和示例丰富

**劣势**:
- ⚠️ 抽象层次高，过度封装
- ⚠️ 学习曲线陡峭
- ⚠️ 调试困难
- ⚠️ 版本更新快，API不稳定
- ⚠️ 对于简单RAG可能过重

**适用场景**: 复杂的多Agent应用，长期维护

**学习成本**: 7-10天

### 3.3 方案C: Haystack

**简介**: deepset开源的NLP框架，专注于RAG。

**优势**:
- ✅ 专注RAG和问答系统
- ✅ 模块化设计，灵活性高
- ✅ 支持多种检索策略
- ✅ Pipeline可视化调试
- ✅ 生产级应用验证

**劣势**:
- ⚠️ 社区相对较小
- ⚠️ 中文资料较少
- ⚠️ 学习曲线中等

**适用场景**: 生产级RAG系统，需要高度定制

**学习成本**: 5-7天

### 3.4 方案D: 自研轻量框架

**简介**: 基于OpenAI SDK和pgvector自研简化版RAG。

**优势**:
- ✅ 代码简洁，易于理解和调试
- ✅ 无框架依赖，轻量灵活
- ✅ 完全可控，易于优化
- ✅ 学习成本低（团队熟悉Python）
- ✅ 适合MVP快速验证

**劣势**:
- ⚠️ 需要自己实现基础功能
- ⚠️ 高级特性需要自行开发
- ⚠️ 缺少现成的最佳实践

**适用场景**: MVP阶段，简单RAG需求

**学习成本**: 1-2周（开发和调试）

**核心代码示例**:
```python
import openai
from pgvector psycopg2 import register_vector

class SimpleRAG:
    def __init__(self, db_conn, embedding_model="text-embedding-3-small"):
        self.db_conn = db_conn
        self.embedding_model = embedding_model
        register_vector(self.db_conn)

    def add_documents(self, documents):
        """添加文档到向量数据库"""
        for doc in documents:
            # 生成embedding
            embedding = openai.Embedding.create(
                input=doc["text"],
                model=self.embedding_model
            )["data"][0]["embedding"]

            # 存储到数据库
            self.db_conn.execute(
                "INSERT INTO documents (text, metadata, embedding) VALUES ($1, $2, $3)",
                (doc["text"], doc["metadata"], embedding)
            )

    def query(self, question, top_k=3):
        """查询相关文档"""
        # 生成问题embedding
        question_embedding = openai.Embedding.create(
            input=question,
            model=self.embedding_model
        )["data"][0]["embedding"]

        # 向量检索
        results = self.db_conn.execute(
            """SELECT text, metadata FROM documents
               ORDER BY embedding <-> $1
               LIMIT $2""",
            (question_embedding, top_k)
        )

        # 构建上下文
        context = "\n".join([r["text"] for r in results])

        # LLM生成答案
        response = openai.Chat.create(
            model="gpt-4",
            messages=[
                {"role": "system", "content": "你是一个法律助手..."},
                {"role": "user", "content": f"根据以下内容回答问题：\n\n{context}\n\n问题：{question}"}
            ]
        )

        return response["choices"][0]["message"]["content"]
```
